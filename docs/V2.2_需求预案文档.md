## 一、背景：v2.1 诊断发现的核心问题

### 1.1 方向摆动实测数据

v2.1 15 分钟诊断监控（30 次采样，30 秒间隔）发现：

| 币种 | 方向分布 | 方向稳定性 | 摆动模式 |
|------|----------|:--:|------|
| ETH | SHORT 100% | 100% | 无摆动，完美稳定 |
| BTC | SHORT 63% / NEUTRAL 37% | 63% | SHORT ↔ NEUTRAL 反复切换 |
| ZEC | LONG 60% / NEUTRAL 40% | 60% | LONG ↔ NEUTRAL 反复切换 |

### 1.2 根因分析

以 BTC 为例，方向判定四条件检查：

```
EMA:  SHORT (EMA20 < EMA60)  ← 固定满足
RSI:  23-28 (超卖区)          ← 不在 LONG(45-70) 也不在 SHORT(30-55)
MACD: SHORT (DIF < DEA)       ← 固定满足
OI:   随行情 ±0.01% 波动       ← 正负交替
```

当 OI 变化为正时：
- LONG 条件：MACD(1) + OI(1) = **2**（不足 3）
- SHORT 条件：EMA(1) + MACD(1) + OI(1) = **3** → **SHORT**

当 OI 变化为负时：
- LONG 条件：MACD(1) = **1**
- SHORT 条件：EMA(1) + MACD(1) = **2**（不足 3）
- → **NEUTRAL**

**结论**：当 RSI 在超卖/超买区时（不属于任何方向的触发区间），OI 的正负微幅波动即能导致方向在 "有方向 ↔ 无方向" 间切换。这不是逻辑 bug，是边界条件过于敏感。

### 1.3 影响评估

| 影响项 | 严重度 | 说明 |
|--------|:--:|------|
| 方向来回切换 | 中 | 15 分钟内 BTC 有 11 次判定为 NEUTRAL，但实际趋势是 SHORT |
| 告警堆积 | 高 | 同一趋势方向因评分微幅变化，每次轮询都产生新告警 |
| 用户信任 | 高 | "刚提示做空，5 秒后又说不确定了"会严重降低信任度 |
| 实盘交易 | 高 | 方向判定不稳定会诱发频繁开平仓 |

## 三、2.2.1 信号方向抖动控制（详细设计）

### 3.1 问题复现

v2.1 实际监测中 BTC 的方向时序（截取）：

```
21:49:25  NEUTRAL (RSI 24.59, OI -0.0029%)
21:50:25  NEUTRAL (RSI 25.18, OI -0.0041%)
21:51:25  SHORT   (RSI 24.48, OI +0.0000%)
21:51:55  SHORT   (RSI 23.59, OI +0.0187%)  ← 成交量放大，评分升至 65
21:52:25  NEUTRAL (RSI 23.79, OI -0.0089%)  ← OI 变负，方向消失
21:52:55  SHORT   (RSI 24.05, OI +0.0070%)  ← OI 变正，方向恢复
21:53:25  NEUTRAL (RSI 25.33, OI -0.0016%)  ← 又消失了
```

**问题本质**：方向判定的 4 条件中，EMA 和 MACD 是大周期信号（相对稳定），RSI 在极端区不投票，OI 的微幅正负波动成为方向判定的"摇摆票"。一个条件的结果变化就足以让方向从 SHORT 翻到 NEUTRAL。

### 3.2 设计目标

| 目标 | 说明 |
|------|------|
| 方向稳定 | 已判定的方向不会被 OI 的微幅波动轻易翻转 |
| 及时响应 | 真正的趋势反转（如 EMA 死叉→金叉）仍然能在合理时间内被识别 |
| 不压制质量 | 方向锁定 ≠ 评分锁定。分数仍实时刷新供用户参考 |
| 静默期 | 方向改变后有一段冷却期，期间不推送新方向提醒 |

### 3.3 三层过滤机制

```
┌──────────────────────────────────────────────┐
│            原始评分 (每 5s)                   │
│    score: 68, direction: SHORT                │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│  第 1 层：趋势缓冲区 (Trend Signal Buffer)    │
│  ┌─────────────────────────────────────┐     │
│  │ 滚动记录最近 12 次方向判定（1 分钟）  │     │
│  │ 输出：多数表决方向                   │     │
│  │ [S,S,N,S,S,N,S,S,N,S,S,S]          │     │
│  │  SHORT=9, NEUTRAL=3                │     │
│  │ → 表决方向: SHORT                   │     │
│  └─────────────────────────────────────┘     │
└──────────────────┬───────────────────────────┘
                   │ 表决方向 ≠ 上次发布方向？
                   ▼
┌──────────────────────────────────────────────┐
│  第 2 层：迟滞带 (Hysteresis Band)            │
│                                              │
│  当前方向: SHORT                              │
│  ┌──────────────────────────────────────┐    │
│  │  放弃区间          NEUTRAL 缓冲       │    │
│  │  score < 35    35 ───────── 55      │    │
│  │  可翻转为LONG    可转为NEUTRAL       │    │
│  │                                      │    │
│  │  SHORT 方向持续要求: score ≤ 35      │    │
│  │  LONG 方向持续要求: score ≥ 55       │    │
│  │  NEUTRAL 区间: 35~55 (无方向)        │    │
│  └──────────────────────────────────────┘    │
│                                              │
│  SHORT → NEUTRAL: 需表决方向=NEUTRAL 连续 3 次│
│  SHORT → LONG :  需表决方向=LONG  连续 4 次  │
│  NEUTRAL → SHORT: 需表决方向=SHORT 连续 3 次  │
│  NEUTRAL → LONG:  需表决方向=LONG  连续 3 次  │
└──────────────────┬───────────────────────────┘
                   │ 方向已改变？
                   ▼
┌──────────────────────────────────────────────┐
│  第 3 层：方向变更冷却 (Direction Cooldown)   │
│                                              │
│  方向变更后，5 分钟内不推送任何新方向提醒      │
│                                              │
│  冷却期间:                                    │
│    - 评分照常更新（用户仍能看到分数变化）      │
│    - 方向变更在冷却结束后才推送               │
│    - 如果冷却期间方向又变回来，不推送         │
└──────────────────────────────────────────────┘
```

### 3.4 针对实测 BTC 数据的效果验证

将 v2.1 监测中 BTC 的方向序列输入三层过滤：

```
原始: N,N,S,S,N,S,N,S,N,S,S,N,S,N,S,N,N,S,S,N,S,N,S,N,S,N,N,S,N,S
缓冲: N,N,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S,S
      (多数表决过滤后，SHORT 占优，NEUTRAL 被压制)
迟滞: 首次 S 确认 → 连续 >3 次 → 发布 SHORT
      后续即使缓冲中混入 N，只要不连续 3 次 → 保持 SHORT
冷却: 发布 SHORT → 5 分钟内不重复推送
```

**效果预估**：v2.1 中 BTC 出现 11 次 NEUTRAL，三层过滤后预期压缩至 0-1 次方向变更提醒。

### 3.5 关键参数

| 参数 | 默认值 | 含义 |
|------|:------:|------|
| `BUFFER_SIZE` | 12 | 趋势缓冲区窗口（12×5s = 60s） |
| `MAJORITY_RATIO` | 0.5 | 多数表决阈值（>50% 即有效） |
| `HYSTERESIS_LOW` | 35 | 迟滞带下限 |
| `HYSTERESIS_HIGH` | 55 | 迟滞带上限 |
| `NEUTRAL_CONFIRM` | 3 | NEUTRAL 确认次数（连续 3 次 = 15s） |
| `DIRECTION_CONFIRM` | 3 | 方向进入确认次数 |
| `FLIP_CONFIRM` | 4 | 翻转确认次数（LONG ↔ SHORT 需连续 4 次 = 20s） |
| `COOLDOWN_MINUTES` | 5 | 方向变更后冷却时间（分钟） |

### 3.6 数据结构

```python
@dataclass
class DirectionTracker:
    """方向跟踪器（每个币种一个实例）"""
    symbol: str
    buffer: deque                      # 最近 N 个方向判定
    published_direction: str = "NEUTRAL"
    published_score: int = 0
    direction_streak: int = 0          # 当前表决方向的连续次数
    streak_target: str = "NEUTRAL"     # 正在累计的方向
    cooldown_until: float = 0.0        # 冷却结束时间戳
    pending_direction: str = None      # 冷却期间待推送的方向

    def feed(self, raw_direction: str, raw_score: int) -> dict:
        """输入原始判定 → 返回 {should_notify, direction, score}"""
```

## 四、2.2.2 告警去重优化

### 4.1 问题描述

v2.1 15 分钟监测中，告警列表严重堆积（超过 20 条）。根因是告警去重键使用 `f"{symbol}:{direction}:{score}:{price}"`，价格每 5 秒微幅变化即产生新键。

### 4.2 方案

```
去重键改为：f"{symbol}:{direction}:{score//5*5}"  # 按 5 分档
并增加：同一 symbol+direction 的告警，冷却期 2 分钟内不重复推送
```

## 五、2.2.3 WebSocket 实时数据推送

### 5.1 目标

- 将 5 秒 REST 轮询替换为 OKX Public WebSocket 频道
- 覆盖：tickers / mark-price / index-price / funding-rate / open-interest
- 降低延迟（WebSocket 推送 < 200ms vs REST 轮询 5s）

### 5.2 技术选型

- `websockets` 库（异步原生支持 FastAPI）
- 频道：`public` 频道（无需鉴权）
- 需要处理的频道：
  - `tickers`：最新价格、24h 涨跌幅
  - `mark-price`：标记价格
  - `index-price`：指数价格
  - `funding-rate`：资金费率
  - `open-interest`：持仓量

### 5.3 架构影响

- 新增 `websocket_service.py`（WebSocket 连接管理）
- `market_service.py` 的事件循环从 APScheduler 定时器改为 WebSocket 回调驱动
- K 线数据仍从 REST 拉取（Public WebSocket 不支持历史 K 线）

    id INTEGER PRIMARY KEY AUTOINCREMENT,
    symbol TEXT NOT NULL,

## 十、与 v2.1 的关系

- v2.2 不修改 v2.1 的评分算法逻辑
- v2.2 在 v2.1 的输出之上叠加"方向滤波层"
- 方向抖动控制是独立模块，可热插拔（不影响 v2.1 评分核心）

---

> **总结**：v2.2 的核心命题是"让信号稳重起来"。  
> 不是在 v2.1 的评分上做加法，而是在方向判定上加一层时序滤波，  
> 让用户看到的方向判定不再是"每 5 秒一惊一乍"，而是持续、连贯、可信任的趋势判断。
