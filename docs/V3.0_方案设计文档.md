## 一、背景与目标

### 1.1 当前系统状态（v2.2）

v2.2 已实现：
- **DirectionTracker** — 方向三层过滤，消除抖动
- **Decision Engine** — 决策追溯 + 置信度 + 趋势阶段
- **SignalDB** — SQLite 历史信号持久化
- **Alert dedup** — 告警去重

v2.2 的核心能力是**分析**，但它不做两件事：
1. 不记录**预测是否正确**（无法评估自身表现）
2. 不根据历史表现**自我优化**（权重固定、无学习能力）

### 1.2 V3.0 要解决的问题

| 问题 | 现状 | V3.0 目标 |
|------|------|-----------|
| 无反馈闭环 | 信号发出后，没人知道是否对 | 自动验证预测结果 |
| 权重固化 | 五维权重是经验值，从不调整 | 根据命中率动态优化 |
| 无经验复用 | 每笔交易都是孤立的 | 案例检索 + 相似状态匹配 |
| 无市场认知 | 只能判断当前方向，不知道处于什么市场 | 牛市/熊市/震荡分类 |

## 三、新增模块详细设计

### 3.1 Prediction Logger

#### 职责
每次 AI Decision 产生时，将完整预测记录持久化，为后续验证和学习提供数据基础。

#### 触发条件
每次信号引擎完成评估（`_refresh_one`），且方向 != NEUTRAL 时触发。

#### 记录字段

| 字段 | 类型 | 说明 |
|------|:----:|------|
| id | INTEGER PK | 主键 |
| symbol | TEXT | 币种 |
| predicted_direction | TEXT | LONG / SHORT |
| predicted_score | INTEGER | 0-100 评分 |
| price_at_prediction | REAL | 记录时的最新价 |
| confidence | INTEGER | 0-100 可信度 |
| trend_phase | TEXT | 趋势阶段 |
| market_regime | TEXT | 市场类型 |
| decision_trace | TEXT(JSON) | 完整决策依据序列化 |
| signal_snapshot | TEXT(JSON) | 当前信号的完整快照 |
| status | TEXT | pending / validated / expired |
| created_at | TEXT | 创建时间 (ISO8601) |

#### 伪代码

```python
class PredictionLogger:
    def log_prediction(symbol, analysis_result, signal):
        if analysis_result.direction == "NEUTRAL":
            return  # 不记录无方向预测
        
        prediction = {
            "symbol": symbol,
            "predicted_direction": analysis_result.direction,
            "predicted_score": analysis_result.score,
            "price_at_prediction": signal.get("price"),
            "confidence": signal.get("confidence"),
            "trend_phase": signal.get("trend_phase", {}).get("phase"),
            "market_regime": signal.get("market_regime"),
            "decision_trace": json.dumps(signal.get("decision_trace")),
            "signal_snapshot": json.dumps(signal),
            "status": "pending",
        }
        conn.execute("INSERT INTO predictions ...", prediction)
```

### 3.2 Future Validator

#### 职责
在预测发出后 30分钟 / 1小时 / 4小时，检查实际价格走势是否与预测一致。

#### 验证逻辑

```
如果是 LONG 预测：
  验证时价格 > 预测时价格 + 阈值 → correct
  验证时价格 < 预测时价格 - 阈值 → incorrect
  否则 → expired (方向未形成)

如果是 SHORT 预测：
  验证时价格 < 预测时价格 - 阈值 → correct
  验证时价格 > 预测时价格 + 阈值 → incorrect
  否则 → expired

阈值 = max(ATR * 0.5, 价格 * 0.001)
```

#### 调度方式
- 在 APScheduler 中注册定时任务，每 5 分钟执行一次
- 查询 `predictions WHERE status='pending'`
- 检查每个预测是否达到验证窗口
- 达到窗口 → 获取当前价格 → 判定结果 → 更新状态

#### 验证结果表

```sql
CREATE TABLE prediction_results (
    id INTEGER PRIMARY KEY,
    prediction_id INTEGER REFERENCES predictions(id),
    validated_at TEXT,
    validation_window TEXT,     -- "30min" / "1h" / "4h"
    price_at_validation REAL,
    actual_direction TEXT,
    is_correct INTEGER,        -- 0 / 1 / -1(expired)
    movement_pct REAL,
    details TEXT(JSON)         -- 验证详情
);
```

### 3.4 Weight Optimizer

#### 职责
根据历史命中率，动态调整五维评分权重，使评分系统持续自优化。

#### 算法设计

```python
class WeightOptimizer:
    # 初始权重（匹配 v2.2 默认值）
    base_weights = {
        "trend": 30,
        "momentum": 20,
        "volume": 20,
        "oi": 20,
        "funding": 10,
    }
    step_size = 1      # 每次调整步长（保守调整）
    min_weight = 5     # 最小权重（防止完全忽略某个维度）
    max_weight = 40    # 最大权重（防止过度依赖）
    
    def optimize(symbol, lookback_days=7):
        """
        1. 获取近 N 天的已验证预测
        2. 计算每个维度的贡献分 vs 实际结果的相关性
        3. 高相关性维度 → 权重 +step_size
        4. 低相关性维度 → 权重 -step_size
        5. 在 min/max 范围内限制
        6. 保存到 weight_history
        """
```

#### 维度命中率计算

每个维度的"命中率" = 该维度贡献的方向与实际方向一致的次数 / 总预测次数

例如：
- trend 维度 70% 预测正确 → 权重维持或 +1
- funding 维度 40% 预测正确 → 权重 -1

#### 调整约束
- **单次调整 ≤ 1 点**（避免突变）
- **每日最大调整 ≤ 3 点**（避免过度反应）
- **权重更改记录到数据库**，前端可回溯

#### 相似度算法

def compute_similarity(current_state, past_case):

### 3.6 Market Regime Classifier

#### 职责
基于技术指标识别当前市场状态，为其他模块提供上下文。

#### 分类体系

| 类型 | 判定条件 | 典型特征 |
|:----:|---------|----------|
| **牛市** | EMA20 > EMA60 > EMA120, EMA 斜率 > 0.1% | 趋势向上，量价配合 |
| **熊市** | EMA20 < EMA60 < EMA120, EMA 斜率 < -0.1% | 趋势向下，持续缩量 |
| **震荡** | EMA20/EMA60 反复交叉, ATR 低 | 无明显方向，窄幅波动 |
| **高波动** | ATR > 过去20日均值 × 1.5 | 剧烈波动，方向不明 |

#### 算法

```python
def classify_regime(indicators: dict) -> str:
    ema20 = indicators.get("ema20", 0)
    ema60 = indicators.get("ema60", 0)
    ema120 = indicators.get("ema120", 0)
    atr14 = indicators.get("atr14", 0)
    avg_price = indicators.get("price", 1)
    atr_pct = atr14 / avg_price * 100
    
    # 趋势判断
    if ema20 > ema60 > ema120:
        return "bull"   # 牛市
    elif ema20 < ema60 < ema120:
        return "bear"   # 熊市
    elif atr_pct > 2.0:
        return "high_volatility"  # 高波动
    else:
        return "ranging"  # 震荡
```

### 3.7 ML Model

#### 职责
使用机器学习模型辅助方向预测，初期采用 XGBoost/LightGBM。

#### 特征工程

| 特征 | 来源 | 说明 |
|------|------|------|
| ema_ratio | EMA20 / EMA60 | 趋势强度 |
| ema_slope_20 | EMA20 斜率 | 短期动量 |
| rsi_normalized | RSI14 / 100 | 归一化 RSI |
| macd_histogram | DIF - DEA | MACD 动能 |
| macd_cross | DIF - DEA (二值) | 金叉/死叉 |
| atr_normalized | ATR14 / EMA20 | 波动率 |
| oi_change | ΔOI | 资金态度 |
| funding_rate | 费率 | 市场情绪 |
| regime | 分类结果 | 市场状态 |
| confidence | v2.2 置信度 | 信号可信度 |

#### 模型训练流程

```
初始阶段（V3.0）：
  无训练，只做特征收集和评估基线

V3.1：
  1. 从 Experience Database 读取已验证案例
  2. 提取特征 + 标签（方向是否正确）
  3. 训练 XGBoost 分类器
  4. 交叉验证评估准确率
  5. 模型持久化存储

预测推理：
  新信号 → 提取特征 → 模型预测 → 输出概率
  与规则引擎结果交叉验证
  不一致时降权处理
```

#### 评估指标
- 方向预测准确率
- 精确率/召回率/F1
- AUC-ROC
- 学习曲线（sample size vs accuracy）

## 七、数据流设计

### 7.1 完整数据流

```
[OKX API 轮询 5s]
    │
    ▼
[Indicator Engine: EMA/RSI/MACD/ATR/OI/费率]
    │
    ▼
[Signal Engine: 五维评分 + DirectionTracker]
    │
    ├──→ [Market Regime Classifier] ──────→ 保存 regime
    │
    └──→ [AI Decision: direction/score/confidence]
            │
            ├──→ [Prediction Logger] ──→ INSERT predictions
            │                               │
            │                               ▼
            │                         [Future Validator: 每5min]
            │                               │
            │                               ├──→ 30min 到期 → 验证 → UPDATE status
            │                               ├──→ 1h 到期 → 验证 → UPDATE status
            │                               └──→ 4h 到期 → 验证 → UPDATE status
            │                                          │
            │                                          ▼
            │                               [Experience Database]
            │                               INSERT experience_cases
            │                                          │
            │                                          ▼
            │                               [Weight Optimizer: 每日]
            │                               UPDATE weights
            │                                          │
            │                                          ▼
            │                               [Case-Based Reasoning: 实时]
            │                               检索相似案例 → 加入分析响应
            │
            └──→ [Frontend: 1s 刷新]
                    │
                    ├──→ MarketCard (价格/指标/标记价)
                    ├──→ ScoreCard (评分/方向/置信度)
                    ├──→ SignalCard (信号/过滤状态)
                    ├──→ PredictionPanel (预测记录/命中率)
                    └──→ Dashboard (ML 面板)
```

### 7.2 时间线示例

```
T+0:   信号发出 → 记录 prediction (status=pending)
T+30m: FutureValidator 检查 → 验证 30min 窗口
       → 正确 → 记录 result(is_correct=1)
       → 错误 → 记录 result(is_correct=0)
T+1h:  FutureValidator 检查 → 验证 1h 窗口
T+4h:  FutureValidator 检查 → 验证 4h 窗口
       → 所有窗口完成 → status=validated
T+1d:  WeightOptimizer 运行
       → 分析近7天命中率 → 调整权重
T+任意: CBR 检索相似案例
       → 返回 top-5 相似 + 命中率
```

## 十、关键原则

1. **机器学习只辅助决策，不直接控制实盘**
   - ML 输出作为参考信号，与规则引擎交叉验证
   - 最终方向仍由 DirectionTracker 决定

2. **所有预测必须可追溯、可解释**
   - 每条预测保存完整 Decision Trace
   - 每个模型推理附带 SHAP 解释

3. **先学习判断能力，再学习收益率**
   - V3.0 专注于方向预测准确率
   - V3.1 才开始优化收益相关指标

4. **保守权重调整**
   - 单次调整 ≤ 1 点
   - 每日最大调整 ≤ 3 点
   - 权重变化可回退、可审计

5. **渐进式实施**
   - 每个模块独立可开关
   - 不影响 v2.2 现有功能
   - 通过 Feature Flag 控制
